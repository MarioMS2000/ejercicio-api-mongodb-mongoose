1- Crear repo nuevo

    1.1 Haz un repositorio nuevo en GitHub, por ejemplo:

        ejercicio-api-mongodb-mongoose

    Luego clónalo:

        git clone https://github.com/TU_USUARIO/ejercicio-api-mongodb-mongoose.git
        cd ejercicio-api-mongodb-mongoose

2- Inicializar Node
    npm init -y

3- Instalar dependencias

    La plantilla usa express, mongoose y dotenv, y el enunciado te pide Morgan. El package.json del repo plantilla también incluye nodemon.

    Instala:

        npm install express mongoose dotenv morgan
        npm install -D nodemon

    Y deja los scripts así:

        {
        "scripts": {
            "dev": "nodemon server.js",
            "start": "node server.js"
        }
        }

4- Variables de entorno

    Servidor → dotenv → variables → config app

    Crea el archivo .env:

        PORT=3000
        MY_MONGO_URI=mongodb://127.0.0.1:27017/ejercicio_api_rest

    Y .env.example:

        PORT=3000
        MY_MONGO_URI=

    La plantilla ya expone MY_MONGO_URI como variable de entorno esperada.

5- Conexión a MongoDB externalizada

    Servidor → config/db_mongo → MongoDB

    config/db_mongo.js

6- Modelado de datos

    Controller → Model → MongoDB → Model → Controller

    Aquí aplicamos la misma idea del demo de Game y Publisher: el producto guarda un ObjectId apuntando al proveedor, y luego se usa populate() para traer datos relacionados.

    models/Provider.js

7- Servicios auxiliares

    Controller → Service → Model → MongoDB → Model → Service → Controller

    Sirve para separar la lógica de acceso a la base de datos del resto de la aplicación.

        Encapsular consultas a la base de datos.
        Separar lógica (arquitectura limpia).
        Reutilizar código.
        Facilitar mantenimiento.
    
    provider.service.js

8- Controladores

    Cliente → Middleware → Ruta → Controller → Modelo → DB
                                   ↓
    Cliente ← Middleware ← Controller ← Modelo ← DB

    Aquí va la lógica de negocio real.
    Decide qué hacer cuando llega una petición (GET, POST, PUT, DELETE).
    
        controllers/providers.controller.js
        controllers/products.controller.js

9- Routes

    Cliente → Middleware → Ruta → Controller

    Aquí es donde conectas todo lo anterior (controllers) con las rutas HTTP
        Es decir: defines qué URL llama a qué función.
            routes/providers.routes.js
            routes/products.routes.js

10- Middlewares

    Middleware normal
        Cliente → Middleware → Ruta → Controller

    notFound (404)
        Cliente → Middleware → Ruta ❌ → notFound → Respuesta

    errorHandler (500)
    
    Cliente → Middleware → Ruta → Controller 💥 → next(error) → errorHandler → Respuesta

    Una función que se ejecuta entre la petición del cliente y la respuesta

    Flujo en tu API
        Cliente → Middleware → Ruta → Controller → Middleware → Respuesta
                middlewares/notFound.js
                middlewares/errorHandler.js

    Server.js (flujo completo REAL)
        Cliente

        server.js

        Crear cuenta (si no tienes)
        Login

    2. Crear proyecto
        New Project
        Nombre: ejercicio-api (por ejemplo)
        Create

    3. Crear cluster
        Build a Database

        Free (M0) ✅
        Create cluster

        ⏳ Espera 1–2 minutos

    4. Crear usuario de base de datos (MUY IMPORTANTE)

        Te va a pedir:

        Username: por ejemplo admin
        Password: algo tipo 1234abcd

        👉 GUÁRDALO, lo necesitarás luego

    5. Permitir acceso a tu IP
        Network Access → Add IP
        Opción fácil:
        0.0.0.0/0

        👉 Permite acceso desde cualquier sitio (vale para el ejercicio)

    6. Obtener la URI de conexión
        Pulsa Connect
        Elige:
        👉 Drivers
        Copia algo como esto:
        mongodb+srv://admin:<password>@cluster0.xxxxx.mongodb.net/?retryWrites=true&w=majority

    7. Modificar tu .env

        Cambia tu .env por esto:

        PORT=3000

        🔥 IMPORTANTE
        Cambia TU_PASSWORD

        /ejercicio_api_rest

        Ctrl + C

        ✅ Resultado correcto

14- Desplegar en RENDER

    1. Antes de entrar en Render

        Asegúrate de que en GitHub ya has subido todo el proyecto y de que tu package.json tenga estos scripts:

            {
            "scripts": {
                "dev": "nodemon server.js",
                "start": "node server.js"
            }
            }

        Render usa un Web Service para este tipo de app Node/Express, y en la configuración del servicio te pedirá, entre otras cosas, el repositorio, el lenguaje, el comando de build y el comando de arranque. Para Node, Render pone como ejemplos típicos npm install para build y npm start para start.

    2. Crear el servicio en Render

        Entra en Render y haz esto:

            New → Web Service → conecta tu repositorio de GitHub.

        Cuando te salga el formulario, rellénalo así:

            Language: Node
            Build Command: npm install
            Start Command: npm start

        Esos valores encajan con la guía actual de Render para apps Node/Express y con la documentación general de Web Services.

    3. Variables de entorno

        En la sección Environment añade al menos esta variable:

            MY_MONGO_URI=mongodb+srv://USUARIO:PASSWORD@cluster0.xxxxx.mongodb.net/ejercicio_api_rest?retryWrites=true&w=majority

        Render recomienda configurar secretos y credenciales como variables de entorno desde el panel del servicio, y permite añadirlas una a una o incluso importar desde un .env.

        Sobre PORT: tu app debe seguir usando process.env.PORT || 3000, que ya lo haces bien. En Render, para Web Services, el puerto HTTP por defecto es 10000, y también puedes definir PORT manualmente si lo necesitas, pero normalmente basta con que tu servidor lea process.env.PORT.

        Tu server.js debería quedarse así en esa parte:

            const PORT = process.env.PORT || 3000;

            app.listen(PORT, () => {
            console.log(`🚀 Servidor corriendo en http://localhost:${PORT}`);
            });

        Eso vale tanto en local como en Render.

    4. Lanzar el deploy

        Después de guardar la configuración y las variables, pulsa desplegar. Render construirá el servicio con npm install y luego lo arrancará con npm start. Si cambias variables de entorno después, Render te deja guardarlas y desplegar de nuevo desde el mismo panel.

    5. Qué URL te dará Render

        Cuando termine, tendrás una URL pública del estilo:

            https://tu-api.onrender.com

        Y podrás probar, por ejemplo:

            https://tu-api.onrender.com/
            https://tu-api.onrender.com/api/providers
            https://tu-api.onrender.com/api/products

        Render asigna a cada Web Service un subdominio onrender.com basado en el nombre del servicio.

    6. Qué revisar si falla

        Los errores más típicos en este punto son estos:

        Si falla al arrancar, revisa que el Start Command coincida con cómo arrancas tu app localmente; Render pone npm start como formato común para Node. Si falla por configuración, revisa si te falta MY_MONGO_URI, porque Render señala que una causa frecuente de fallos es la ausencia de variables de entorno necesarias.

        Además, como tu base de datos está en Atlas, confirma en MongoDB Atlas que has permitido acceso de red desde Render; si no, tu app puede desplegar pero no conectar con la base.

    7. Lo que entregarás al final

        Cuando esté desplegado, tendrás que pasar:

            el link del repositorio GitHub
            el link de la API en Render

        Por ejemplo:

            Repositorio:
                https://github.com/tuusuario/ejercicio-api-mongodb-mongoose

            API:
                https://tu-api.onrender.com

        crear Web Service en Render
